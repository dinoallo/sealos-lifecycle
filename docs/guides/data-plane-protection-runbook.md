# Data Plane Protection Runbook

## Status

Current operations contract

## Purpose

This runbook defines how to protect Secrets, PVCs, database resources, and
runtime-generated state when operators run `sync revert`, controller rollback,
upgrades, or derived distribution line switches.

The rule is simple: desired-state reconciliation can repair configuration, but
it must not silently replace, delete, or promote data-plane state. Treat
Secret bytes, PersistentVolumeClaim data, database contents, generated
connection Secrets, and backup artifacts as cluster-local runtime state unless
a package explicitly declares a reviewed data migration workflow.

## Data Plane Surfaces

Use this ownership map before any mutating command:

| Surface | Examples | Default Ownership | Allowed Automation |
| --- | --- | --- | --- |
| Secret references | Secret names in values files, `secretRef`, TLS Secret names | local input | render/apply can reference names |
| Local Secret manifests | `local-repo/resources/secrets/*.yaml` | local overlay | apply and scoped revert after owner approval |
| External secret references | `ExternalSecret`, `SecretProviderClass`, store refs | local overlay | apply references, never copy provider payloads |
| Runtime-generated Secrets | KubeBlocks account Secrets, operator-created connection Secrets | runtime-local | observe only |
| PVCs and PVs | `PersistentVolumeClaim`, retained `PersistentVolume` | data-plane runtime | observe and protect from deletion |
| Database resources | KubeBlocks `Cluster`, PostgreSQL data, backup schedules | package baseline plus runtime data | apply spec changes only after backup gate |
| Backup artifacts | snapshots, dump files, object-store backup ids | data-plane evidence | archive references, not credentials |

## Preflight Before Mutation

Run this checklist before `sync revert`, `sync apply`, controller rollback, or a
derived line switch touches a stateful component:

1. Identify the affected objects with `sync status` and `sync diff`.
2. Classify each object as configuration, local Secret material, PVC, database
   control-plane object, or runtime-generated state.
3. Run `sealos sync validate` against the BOM and local repo so Secret manifest
   permissions and local-resource rules are checked.
4. Capture current Kubernetes state:

   ```bash
   kubectl get secret,pvc,pv -A -o yaml
   kubectl get cluster.apps.kubeblocks.io -A -o yaml
   kubectl get events -A --sort-by=.lastTimestamp
   ```

5. Confirm that a recent backup or snapshot exists for every affected database
   or PVC owner. Record only backup ids, timestamps, namespaces, object names,
   and storage locations that do not include credentials.
6. Confirm that the package owner and cluster owner agree on the action when
   the mutation crosses ownership boundaries.

Do not continue when Secret payloads, kubeconfigs, private keys, or provider
tokens would be copied into incident artifacts.

## Operation Gates

| Operation | Required Gate | Block When |
| --- | --- | --- |
| `sync revert` on a Secret | Owner approval for the local Secret manifest or external reference. | The live Secret was generated by an operator or contains rotated credentials that are not in local repo. |
| `sync revert` on PVC or database objects | Backup evidence plus package owner approval. | The revert would delete PVCs, change retention policy, or downgrade storage/database version. |
| Controller rollback | Last successful revision exists and stateful components have backup evidence. | The rollback changes database topology, storage class, Secret references, or PVC templates without a migration note. |
| Upgrade | Package acceptance evidence plus data migration or compatibility notes. | Required backups are missing, healthcheck hooks do not cover the database, or the upgrade changes irreversible storage settings. |
| Derived line switch | Diff of package replacements plus runtime-state compatibility check. | The derived BOM changes database ownership, Secret names, PVC names, or storage policy without a transfer plan. |

## Secrets

Secret names and references can be part of rendered desired state. Secret bytes
remain local.

Allowed:

- applying a local Secret manifest that is intentionally stored under
  `local-repo/resources/`
- applying an `ExternalSecret` or `SecretProviderClass` reference
- reverting a local Secret object to the local repo manifest after the secret
  owner confirms that the recorded value is still valid

Blocked:

- committing runtime-rotated Secret bytes back into a package artifact
- archiving `data`, `stringData`, kubeconfig contents, provider tokens, or
  private keys
- treating an operator-generated connection Secret as package-owned desired
  state

Verification:

```bash
sealos sync status --cluster <cluster> --bundle-dir <bundle> --kubeconfig <kubeconfig> --host-root /
kubectl -n <namespace> get secret <name> -o jsonpath='{.metadata.name}{"\n"}'
```

Only verify identity, type, labels, annotations, owner references, and expected
consumer health. Do not print Secret payloads.

## PVCs And Databases

PVCs, PVs, and database contents are data-plane state. The package can own the
database control-plane manifest, but live data is not a package baseline.

Before changing a database package, PVC template, storage class, retention
policy, topology, or service version:

1. Capture the database custom resource and PVC list.
2. Confirm backup freshness and restore scope.
3. Confirm package release notes or migration notes for the version change.
4. Run package healthcheck hooks or the application acceptance workflow.
5. Record the previous and target BOM revision, desired digest, and package
   digests.

After the change:

1. Confirm the database custom resource is Ready according to its operator.
2. Confirm PVCs still exist and are bound to the expected storage class.
3. Confirm application healthcheck hooks pass.
4. Confirm `sync status` no longer reports unexpected drift for the stateful
   component.
5. Attach backup id, restore instructions, and post-change health evidence to
   the ticket.

## Revert And Rollback

Use `sync revert` for configuration drift only after classifying the affected
surface. Use controller rollback to restore the last successful rendered
revision only when the data-plane compatibility gate passes.

Do not use broad revert or rollback as a database restore mechanism. If data is
corrupted, follow the database owner restore procedure and then reconcile
configuration once the runtime state is stable.

For rollback holds:

- archive the failed target YAML, controller logs, events, bundle path, and
  applied revision path
- keep the last successful bundle and current failed bundle digests
- record whether the rollback touched Secret references, PVC templates,
  storage classes, database versions, or backup schedules
- require explicit owner approval before selecting the next revision

## Upgrades And Derived Line Switches

Treat upgrades and derived line switches as package identity changes plus
runtime compatibility checks.

Before switching:

- compare the source and target BOM revisions
- list changed component packages and digests
- identify stateful package changes
- verify Secret reference names remain compatible or have a planned rotation
- verify PVC names, storage classes, and retention policies remain compatible
- verify database service versions and migration requirements

After switching:

- capture target status or `sync status`
- run application and database healthchecks
- confirm the old backup can still be used for the agreed restore window
- archive the derived BOM labels that identify the source line and revision

If the derived line intentionally changes ownership of a stateful component,
write a handoff note that names the old owner, new owner, protected objects,
backup evidence, restore owner, and validation command.

## Evidence To Keep

Every data-plane-sensitive ticket should keep:

- `sync status -o yaml` and `sync diff -o yaml`
- target or controller status after mutation
- affected object identity list, without Secret payloads
- previous and target BOM revision plus desired-state digests
- package digests for changed stateful components
- backup or snapshot id and timestamp
- database or application healthcheck output
- owner approvals for Secret, PVC, database, or derived-line changes
- rollback or restore instructions

The evidence should prove that the operation changed the intended configuration
without losing or promoting runtime data-plane state.
